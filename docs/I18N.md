# V-Lake · i18n workflow

**Source of truth:** `apps/web/src/lib/i18n/locales/en.json`.
**Target locales:** `cs.json`, `de.json`.
**Runtime:** `usePrefs()` auto-detects the browser locale on first visit
(via `navigator.languages`) and falls back to English. The user can
override it in the Settings menu — that flips `langSource` to `user` and
future auto-detects respect the choice.

## Adding or editing a translation

1. **Add the key to `en.json` first.** English is the reference — every
   locale must expose the same set of scalar keys and the same plural
   families (see below). The English string is what shows if any
   target locale is missing the key at runtime (`translate()` falls
   back to English before the raw key).

2. **Add the equivalent key to `cs.json` and `de.json`.** Two options:
   - **Manual:** copy the key, translate by hand. Recommended for
     branded copy (headlines, tone-of-voice words like "Cold-plunge").
   - **DeepL-assisted:** run `pnpm --filter web i18n:sync` (needs
     `DEEPL_API_KEY` env; free DeepL account gives 500k chars/month).
     The script translates every key missing in a target locale,
     preserving `{placeholder}` tokens through DeepL's XML tag-handling
     mode. Do a human pass over the diff before committing — DeepL is
     excellent for cs/de but doesn't know product-specific idioms.

3. **Run `pnpm --filter web i18n:check`** to confirm the three locales
   are structurally aligned. The same check runs inside the vitest
   suite (`src/lib/i18n/consistency.test.ts`) so CI catches
   half-authored translations before merge.

4. **Use the key in a component.**
   - Client components: `const t = useT(); … t("my.key")`.
   - Client plurals:    `const p = useP(); … p("my.key", count)`.
   - Server components: render a client-boundary island
     (`<TLabel tKey="my.key" />` or `<PLabel baseKey="my.key" count={n} />`
     from `@/components/ui`). See `SectionHeading`, `HeroCopy`,
     `StatLabel`, `RegionHeading` for examples.

5. **Runtime helpers must not return English text.** Pure business
   logic (`assessSwim`, `relativeTime`, `conditionFromCode` when we
   port it) returns i18n **keys** — the UI translates them. This keeps
   the logic locale-neutral, unit-testable in Node, and free of React
   imports.

## Plural handling

Uses `Intl.PluralRules`. A plural family looks like this:

```json
{
  "countries.countLake_one": "{n} lake",
  "countries.countLake_other": "{n} lakes"
}
```

Czech uses three forms:

```json
{
  "countries.countLake_one":  "{n} jezero",
  "countries.countLake_few":  "{n} jezera",
  "countries.countLake_other":"{n} jezer"
}
```

Rules:
- `_other` is **mandatory in every locale** (the CLDR fallback).
- A locale MAY define additional CLDR categories (`_zero`, `_two`,
  `_few`, `_many`) if its grammar uses them. The check-script and the
  vitest test allow this — they compare **plural stems** and require
  `_other` per stem, not the exact category set.

## Placeholders

Curly-brace tokens (`{n}`, `{name}`, `{year}`, `{ago}`) must be
preserved verbatim in every locale. The consistency test scans every
translated string and errors out if a placeholder used by the English
string is missing from a target locale — the DeepL sync script wraps
tokens in `<ph>` XML tags before calling the API so DeepL leaves them
alone.

## Adding a fourth locale

1. Copy `en.json` to `es.json` (or wherever) and translate — or run
   `DEEPL_API_KEY=… node scripts/i18n-sync.mjs --locale=es`.
2. Add the locale to:
   - `SUPPORTED_LOCALES` in `src/lib/i18n/config.ts`
   - `DICTS` in `src/lib/i18n/index.tsx`
   - `LANG_META` in `src/components/layout/settings-menu.tsx` (name, flag)
   - `TARGETS` array at the top of both scripts under
     `apps/web/scripts/i18n-*.mjs`.
3. Add the locale to the vitest consistency test so CI enforces
   parity.
4. Run `pnpm i18n:check` and `pnpm test`. Both must be green before
   commit.

## Automation options considered

| Service | Fit | Notes |
|---|---|---|
| **DeepL API Free** | ★★★ Chosen | 500k chars/mo free, best-in-class quality for cs/de. `Authorization: DeepL-Auth-Key …:fx`. |
| DeepL API Pro | ★★ | Same quality, higher limits (€5.49/mo). Switch by dropping the `:fx` suffix. |
| Google Translate | ★ | Placené, kvalita horší než DeepL pro EU jazyky. |
| OpenAI GPT-4o-mini | ★★ | Nice for context-aware idioms; requires more prompt engineering than a script. Fallback if DeepL down. |
| Weblate / Crowdin / Lokalise | ★ | SaaS overhead; only worth it beyond ~5 locales. |
| Argos Translate (offline) | ★ | Free/open-source but noticeably lower quality; use only if we cannot ship an API key. |

## Files to touch when authoring a new component

If the new component renders any human-readable text, the checklist is:

- [ ] Extract every string into a key inside `en.json`
- [ ] Add the same key to `cs.json` + `de.json` (manually or via
      `i18n:sync`)
- [ ] `pnpm --filter web i18n:check` returns OK
- [ ] `pnpm --filter web test` returns green (the consistency test is
      what catches drift in CI)
- [ ] For server components: use `<TLabel />` / `<PLabel />` islands so
      you don't accidentally turn a route into a client subtree.
- [ ] For components that render currency, temperature, distance —
      check `usePrefs` before hardcoding units. Unit conversion is in
      `src/lib/prefs.ts` (`toDisplayTemp`, `unitSymbol`).

Anything user-facing that is *not* wrapped in `t()/p()` / `<TLabel/>` /
`<PLabel/>` is a bug — it will render in English even for a Czech user
who set the language explicitly. Grep the diff for `>[A-Z]` and quoted
capital-first strings before opening a PR.
